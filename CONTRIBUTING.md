# Contributing to TabyaOS

Thank you for your interest in contributing. TabyaOS is a security-focused project — every change that affects hardening controls must be traceable to a named framework requirement.

## Before You Start

- Read [CLAUDE.md](CLAUDE.md) for project context and working agreements
- Check the [GitHub Issues](../../issues) for existing discussions
- For significant new features or controls, open an issue first to discuss scope

## Development Setup

```bash
git clone https://github.com/tabyaos/tabyaos.git
cd tabyaos
just build-molecule-runner  # build Docker test runner (one-time)
just test-molecule-role cis-l2  # verify your setup works
```

## Contribution Types

### Adding a New Hardening Control

1. Identify the Ansible role that owns the control area (or create a new role)
2. Add the task to the appropriate `tasks/*.yml` file
3. Reference the framework requirement in the task name: `# CIS AL2023 5.2.3` or `# PCI-DSS v4.0 Req 10.2.1.1`
4. Add a verify assertion in `molecule/default/verify.yml` that tests the actual hardened state value (not just file existence)
5. Add an entry to `compliance/control-mappings.yaml`
6. Run `just gen-docs` to regenerate compliance documentation
7. Run `just test-molecule-role <role>` — both converge AND idempotence must pass

### Fixing a Bug

1. Reproduce the bug with a failing molecule test
2. Fix the bug
3. Confirm the test passes
4. Reference any relevant CVE or issue in the commit message

### Improving Documentation

- `docs/` — architecture, getting-started, threat-model, etc.
- `compliance/` — framework mapping files (human-readable markdown)
- Do not edit `compliance/generated/` — these are auto-generated by `just gen-docs`

## Code Standards

- **Ansible**: All tasks must be idempotent (molecule idempotence check enforced in CI)
- **Framework references**: Every hardening task name must include its requirement ID
- **No service management in Docker tests**: Use `manage_services: false` pattern
- **Handlers**: All handlers must have `failed_when: false` for Docker compatibility
- **Variables**: Defaults in `defaults/main.yml`; service flags follow `<role_prefix>_manage_services` pattern
- **ansible-lint**: Must pass in CI

## Pull Request Process

1. Branch from `main`: `git checkout -b feat/your-feature`
2. Commit with descriptive messages referencing framework requirements
3. Ensure `just test-molecule-role <role>` passes (converge + idempotence + verify)
4. Open a PR with:
   - What control was added / changed
   - Which framework requirement it addresses
   - Test evidence (molecule output summary)
5. A maintainer will review within 5 business days

## Commit Message Format

```
<type>(<scope>): <short description>

# Types: feat, fix, test, docs, refactor, ci
# Scopes: cis-l2, pci-dss-v4, aide, fips-mode, etc.

# Example:
feat(pci-dss-v4): enforce sudo logfile via Req 7.2.2

Added Defaults logfile=/var/log/sudo.log to /etc/sudoers.
Molecule verify asserts the logfile is configured.
```
